**Last Updated**: 2025-12-15

## Architecture Pattern

**Primary Pattern**: Clean Architecture + DDD (Library-First) for CLI

Python 製の単一リポジトリ。ドメインモデルを中心に据え、ユースケース（アプリケーションサービス）がドメインを呼び、インフラはポートを実装する。CLI はインターフェース層として薄く保つ。

---

## Architecture Layers (Language-Agnostic)

### Layer 1: Domain / Core

**Purpose**: 音楽メタデータのモデルとビジネスルール
**Rules**:
- 他のレイヤに依存しない
- エンティティ: Track, Album, Artwork, TagSet
- 値オブジェクト: FilePath, TagValue
- ドメインサービス: タグ検証、正規化など（I/O なし）

### Layer 2: Application / Use Cases

**Purpose**: ユースケースのオーケストレーション
**Rules**:
- Domain のみ依存
- ポート/DTO を定義し、I/O はインフラに委譲
- 例: ScanDirectory, ApplyMetadata, RenameFiles, ValidateTags

### Layer 3: Infrastructure / Adapters

**Purpose**: 外部との接続実装
**Rules**:
- Application のポートを実装
- ファイルシステム I/O、Mutagen によるタグ入出力、CSV/TSV 読み書き、将来のオンライン API クライアント

### Layer 4: Interface / Presentation

**Purpose**: エントリポイント（CLI）
**Rules**:
- Application のユースケースを呼び出す薄い層
- 入力バリデーション、ヘルプ、出力整形のみ担当

### Layer Dependency Rules

```
Interface (CLI)
  ↓
Application (Use Cases)
  ↓
Infrastructure (Adapters)
  ↓
Domain (Core)
```

依存は外側から内側のみ。Domain は他に依存しない。

## Library-First Pattern (Article I)

コア機能は `src/music_metadata_lib` に置き、`src/music_metadata_tool` は配布用 CLI の薄い入口として扱う。インフラ実装が増えてもドメイン/アプリケーションは純粋ロジックを維持し、テスト可能性を確保する。

---

## Application Structure (CLI)

CLI ルートコマンド（仮: `music-metadata`）が提供する想定:
- `scan`: ディレクトリ走査と CSV/TSV 出力
- `apply`: CSV/TSV からタグを書き戻し（ドライラン/実行）
- `rename`: タグに基づくファイル名/ディレクトリ整形
- `validate`: 必須タグ/エンコードチェックのレポート
- `config`: プロファイル/テンプレート管理

各コマンドは Application 層のユースケースに対応し、I/O は Infrastructure 層で実装する。

## Testing Strategy

- **Unit**: Domain と Application のロジックを pytest でカバー
- **Integration**: Mutagen を使ったタグ読み書き、CSV/TSV I/O、ファイルシステム操作
- **End-to-End**: CLI コマンドのドライラン/実行をサンプルデータで検証
- **Performance Smoke**: 1,000 ファイル規模の処理時間/メモリ使用量を確認

## Deployment/Distribution

- OSS 配布: PyPI 配布を視野に入れ、`pipx` でインストール可能にする
- コンテナ: Dockerfile/compose で開発環境を再現
- CI: GitHub Actions で `ruff check`, `ruff format --check`, `mypy`, `pytest` を実行
