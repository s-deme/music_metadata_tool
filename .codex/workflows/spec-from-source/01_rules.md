# Reverse Spec Rules

以下のルールを厳守してリバース仕様化を行う。

1. コードで確認できない内容は書かない。
2. 推測で穴埋めしない。不明な点は `要確認` と明記する。
3. 省略しない。条件分岐、エラー、前提条件も拾う。
4. 利用者視点で書く。ただし根拠は実装に置く。
5. 1つの記述に複数機能を混ぜない。
6. 入力、処理、出力、失敗時挙動を分けて捉える。
7. 根拠となるファイルまたはテストを必ず残す。
8. README と実装が衝突したら実装を優先し、差分を備考に残す。
9. テストで保証されていない挙動は「実装確認のみ」と分けて扱う。

## 禁止事項

- 「おそらく」「通常は」「多分」のような曖昧表現
- 実装されていない将来構想の混入
- 1行に複数の条件や例外を詰め込む記述

## 記述の粒度

- CLI では「コマンド単位」より細かく、「オプション・出力・エラー条件」まで分解する
- 内部実装のクラス説明ではなく、利用者が観測できるふるまいを主に書く
